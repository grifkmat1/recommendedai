# RecommendedAI

**Recommendation engine comparing three algorithmic approaches** — end-to-end analytics project demonstrating user behavior modeling, collaborative filtering, content-based filtering, and offline evaluation, with a FastAPI serving layer.

---

## Executive summary

RecommendedAI is a production-patterned recommendation system that ingests user interaction events (views, clicks, purchases), builds a weighted user–item interaction matrix, and compares three distinct recommendation strategies — popularity baseline, item-based collaborative filtering, and category-based content filtering — using a rigorous leave-last-out offline evaluation protocol.

The project is structured to mirror a real analytics and engineering workflow: raw event data → feature engineering → model comparison → metric reporting → REST API deployment. It targets the core business question every product team faces: *which recommendation strategy drives the most relevant results for users, and how do we measure that objectively?*

---

## Business problem

E-commerce and content platforms generate thousands of user interaction events daily. Without a recommendation layer, users must manually browse catalogs — increasing friction and reducing both session depth and conversion rate. The key challenges this project addresses:

- **Discovery problem** — most items in a catalog go unseen; recommendation surfaces relevant items users would not find organically.
- **Personalization vs. scale** — simple popularity-based recommendations are easy to implement but treat all users identically; collaborative filtering personalizes at scale.
- **Evaluation rigor** — recommending items that users would have found anyway provides no business value; offline evaluation must simulate real prediction conditions to be meaningful.
- **Cold-start tradeoff** — new or inactive users have sparse interaction histories; a fallback strategy (popularity) is required when personalized signals are insufficient.

---

## Business value

Recommendation systems directly drive measurable business outcomes:

| Business outcome | Mechanism |
|---|---|
| **Increased session depth** | Relevant recommendations keep users browsing longer, reducing bounce rate |
| **Higher conversion rate** | Surfacing items aligned with demonstrated purchase intent increases add-to-cart and checkout events |
| **Catalog utilization** | Personalized recommendations expose long-tail inventory; popularity-only recommendations concentrate traffic on top items |
| **Retention improvement** | Users who receive relevant recommendations return more frequently — recommendations become a habit-forming product feature |
| **Decision support** | Hit-rate@K metrics give product teams an objective framework for comparing and A/B testing recommendation strategies before production rollout |

The **catalog coverage** metric (measured in this project) captures whether a recommendation strategy serves the whole catalog or concentrates on a few popular items — a key concern for marketplace and inventory planning teams.

## Recommendation approaches: methods and tradeoffs

### 1. Popularity baseline
Recommends the globally most-interacted items across all users.

**Business rationale:** The honest starting point. Every recommendation system should be compared against "what would we show everyone?" before claiming personalization adds value. Popularity is always deployable (no cold-start problem) but ignores individual user context entirely.

**Limitation:** Concentrates 100% of recommendations on 6.2% of the catalog. Users with niche interests receive irrelevant recommendations. Drives rich-get-richer dynamics in item visibility.

### 2. Item-based collaborative filtering (Item-CF)
Builds an item–item cosine similarity matrix from the weighted interaction matrix, then scores unseen items by their similarity to each user's interaction history.

**Business rationale:** Item similarities are more stable over time than user similarities — adding one new user doesn't require recomputing the whole item space. This makes Item-CF operationally practical at scale. Personalizes to each user's demonstrated preferences without requiring demographic or content data.

**Limitation:** Requires a minimum interaction history per user. New users with no events receive no signal (cold-start). Addressed here with a popularity fallback in the API.

### 3. Category-based content filtering
Identifies each user's most-engaged product category, then recommends popular unseen items within that category.

**Business rationale:** Uses the item catalog's structured metadata (category) as a signal when collaborative data is sparse. Particularly useful for new users who have a few interactions — even one purchase in "electronics" provides a category signal. Interpretable and easy to explain to business stakeholders.

**Limitation:** Category is a coarse signal — "electronics" covers headphones, laptops, and cables, which may be very different purchase intents. Performance ceiling is lower than CF at higher K values (HR@10: 0.360 vs 0.340 — competitive, but diverges as K grows).

### Why leave-last-out?
Holding out each user's *final* interaction simulates the production scenario: the model must predict what a user will interact with *next*, trained only on their past. This is temporally honest. Random hold-out would allow the model to "predict" past interactions using future data — overstating real performance.

## Technical architecture

```
events.csv  (2,860 interactions: user_id, item_id, event_type)
items.csv   (80 items: item_id, category)
        │
        ▼
   data.py  ──►  Event weighting
                 purchase × 4  ·  click × 2  ·  view × 1
                 ──►  Weighted user × item matrix (200 × 80)
        │
        ├──► recommenders.py
        │    ├── popularity_recommender()     Global top-K
        │    ├── item_cf_recommender()        Cosine similarity on item vectors
        │    └── category_recommender()       Top-K within user's fav category
        │
        ├──► evaluate.py
        │    Leave-last-out split → hit_rate_at_k() across all 3 approaches
        │
        └──► api.py
             FastAPI  →  GET /recommend/{user_id}
                         GET /health
                         Cold-start fallback: popularity for unknown users
```

### Event weighting rationale

Purchase signals are assigned 4× the weight of views because they represent completed intent — a user who purchases an item is providing a far stronger preference signal than one who views it. Click weight (2×) falls between: it indicates active interest without full commitment. This weighting scheme is standard in implicit feedback recommendation systems.

## Using real interaction data

To evaluate on real data:

1. Prepare events in the schema: `user_id, item_id, event_type` (event_type values: `view`, `click`, `purchase`)
2. Prepare items: `item_id, category`
3. Drop files into `data/` replacing the synthetic CSVs
4. Run: `PYTHONPATH=. python src/evaluate.py`

On real e-commerce data with sufficient history, expect HR@5 in the 0.05–0.20 range for Item-CF — the synthetic dataset's numbers are higher because the data was generated with intentional category preferences.

### PostgreSQL integration
Replace CSV files with a PostgreSQL backend:
- `events` table: `user_id, item_id, event_type, timestamp`
- `items` table: `item_id, category, name, price`
- `recommendations` table: cache generated recs with TTL

### AWS deployment architecture
Target architecture for cloud deployment:
- **ECR** — container registry for Docker images
- **ECS Fargate** — serverless container hosting, no EC2 management
- **RDS PostgreSQL** — managed database for events and items
- **Application Load Balancer** — HTTPS termination and routing
- **CloudWatch** — API latency, error rate, and recommendation coverage monitoring

### Additional evaluation metrics
- **Precision@K** and **Recall@K** (complement to hit-rate)
- **NDCG@K** (normalized discounted cumulative gain — rewards ranking the right item higher)
- **Online A/B testing** framework (compare strategies on live traffic)
- **Serendipity** and **diversity** metrics (measure recommendation variety)

### CI/CD improvements
- Add Docker image build and ECR push to GitHub Actions workflow
- Add staging environment deployment step
- Add recommendation quality regression test (alert if HR@5 drops below threshold)
